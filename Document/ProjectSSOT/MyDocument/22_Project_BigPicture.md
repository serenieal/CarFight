# CarFight 큰 그림

> 역할: **문서 운영용이 아니라, 프로젝트를 한눈에 이해하기 위한 사용자용 개요 문서**  
> 마지막 정리(Asia/Seoul): 2026-03-20

---

## 이 문서는 왜 필요한가
지금 `Document/ProjectSSOT/` 안의 문서들은 대부분 **작업을 끊어서 관리하기 좋게** 만들어져 있다.

그래서 장점은 있다.
- 현재 해야 할 일 찾기 쉽다.
- 검증 기준 찾기 쉽다.
- 세부 결정 추적이 쉽다.

하지만 단점도 있다.
- 프로젝트를 전체적으로 보면 어떤 게임/시스템을 만들고 있는지 한 번에 안 들어온다.
- 지금 왜 이 작업을 하는지, 그 작업이 전체 구조에서 어디에 붙는지 흐려진다.
- 문서를 다 읽어도 "그래서 지금 이 프로젝트의 핵심은 뭐지?"가 남을 수 있다.

이 문서는 그 문제를 줄이기 위해 만든다.

즉, 이 문서는 **당장 작업하기 위한 문서**가 아니라,
**이 프로젝트를 머릿속에 한 장으로 넣기 위한 문서**다.

---

## 한 줄 요약
CarFight는 현재 문서 기준으로 보면,
**차량 플레이를 중심으로 한 프로젝트이며, 지금 개발의 중심은 `BP_CFVehiclePawn`를 겉껍데기(Thin BP)로 두고, 실제 판단과 상태는 C++ 코어로 옮겨서 안정적인 차량 코어를 만드는 것**이다.

조금 더 풀면 이렇다.
- 플레이어가 조작하는 현재 기준 차량이 있다.
- 그 차량은 이미 BP 중심 실험 단계가 아니라 **C++ 중심 구조**로 정리되고 있다.
- 지금 가장 중요한 것은 새로운 기능을 마구 늘리는 것이 아니라,
  **주행 상태 판단(DriveState)** 과 **휠 시각 동기화(Wheel Sync)** 를 현재 구조에서 안정적으로 굴리는 것이다.
- 이후의 세부 차량 감각 조정, 카메라 판단, 데이터 확장 등은 이 코어가 안정된 다음 단계다.

---

## 지금 프로젝트의 실제 중심축
현재 문서에서 가장 강하게 보이는 중심축은 5개다.

### 1) 플레이 축
- 현재 기준 플레이 차량은 `BP_CFVehiclePawn`이다.
- 예전 기준이었던 `BP_ModularVehicle`는 더 이상 현재 기준이 아니다.
- 즉, 지금 플레이 기준선은 이미 새 차량 Pawn 쪽으로 넘어왔다.

### 2) Native 코어 축
- 실제 기준 Native Pawn은 `ACFVehiclePawn`이다.
- 이 Pawn이 차량 코어의 중심 몸통 역할을 한다.

### 3) 주행 판단 축
- `UCFVehicleDriveComp`가 현재 핵심이다.
- 이 컴포넌트는 단순 입력 전달자가 아니라,
  **지금은 주행 상태를 판단하는 코어**로 보는 것이 맞다.
- 즉, 앞으로 차량이 어떤 상태인지 판단하는 기준은 여기서 나온다.

### 4) 휠 시각 동기화 축
- `UCFWheelSyncComp`가 휠의 시각 동기화를 맡는다.
- 예전에는 BP 실험, helper, debug pipe, 단일 축 테스트 같은 과도기 절차가 많았지만,
  지금은 그 실험 기록을 정리하고 **최종 기준 구조만 남기는 단계**다.

### 5) 데이터 축
- `UCFVehicleData`가 차량 데이터를 들고 가는 축이다.
- 현재는 단순 값 저장소가 아니라,
  **차량별 설정을 DataAsset로 관리하고 C++ 코어에 적용하는 방향**으로 이해하면 된다.
- 테스트 기준 자산은 현재 `DA_PoliceCar`다.

## 지금 구조를 사람 말로 풀면
현재 구조를 기술 이름 없이 사람 말로 바꾸면 아래처럼 이해하면 된다.

### `BP_CFVehiclePawn`
- 플레이어가 실제로 보게 되는 차량 껍데기
- 화면에 보이는 표현, 조립, 반응을 담당
- 하지만 복잡한 규칙 판단은 여기서 하지 않음

### `ACFVehiclePawn`
- 차량의 본체
- 게임 안에서 차량이 어떻게 존재하는지의 기준 몸통

### `UCFVehicleDriveComp`
- 이 차가 지금
  - 정지인지
  - 가속 중인지
  - 관성 주행인지
  - 제동 중인지
  - 후진 중인지
  - 공중 상태인지
  같은 것을 판정하는 핵심 두뇌

### `UCFWheelSyncComp`
- 바퀴가 화면에서 어떻게 움직여 보여야 하는지 정리하는 축
- 조향, 회전, 서스펜션 표현이 여기에 연결된다.

### `UCFVehicleData`
- 차량별로 달라지는 데이터 묶음
- 예를 들어 차량 느낌, 주행 상태 판정값, 각종 설정이 여기에 모일 방향이다.

---

## 왜 지금 C++ 중심 구조를 고집하나
현재 문서에서 일관되게 보이는 결정은 이것이다.

> **판단 / 상태 / 규칙은 C++에 두고, BP는 Thin BP로 유지한다.**

이 결정이 중요한 이유는 간단하다.

### BP에 판단을 많이 넣으면 생기는 문제
- 규칙이 여기저기 흩어진다.
- 상태가 어디서 바뀌는지 찾기 어렵다.
- 디버깅이 힘들다.
- 프로젝트가 커질수록 "왜 이렇게 됐는지"를 설명하기 어려워진다.

### C++ 중심으로 두면 얻는 것
- 상태 판정 기준을 한곳에 모을 수 있다.
- 테스트 기준을 문서로 고정하기 쉽다.
- BP는 표현만 담당하므로 그래프가 비대해지지 않는다.
- 나중에 기능이 늘어나도 기준선을 유지하기 쉽다.

그래서 지금 단계는
**“BP에서 C++로 옮기는 중” 자체보다**
**“이제 C++ 중심 코어를 흔들지 않고 유지하는 단계”**
로 보는 게 맞다.

---

## 지금 가장 중요한 개발 주제: DriveState
현재 문서들을 관통해서 보면, 지금 가장 중요한 개발 주제는 **DriveState**다.

DriveState는 쉽게 말해,
차가 현재 어떤 상태인지 게임이 이해하는 공식 언어다.

예:
- `Idle`
- `Accelerating`
- `Coasting`
- `Braking`
- `Reversing`
- `Airborne`

이게 왜 중요하냐면,
차량 게임에서 많은 시스템이 결국 이 상태를 기준으로 움직이기 때문이다.

예를 들면:
- HUD나 디버그 표시
- 사운드 반응
- 카메라 반응
- 애니메이션/연출 반응
- 후속 게임플레이 판정

현재 단계에서는 이 DriveState를
- **차종별 감각 조정**보다 먼저
- **공통 코어 기준으로 안정화**하는 것이 우선이다.

즉 지금 질문은
"경찰차 느낌을 완벽히 만들었나?" 보다,
"어느 차량이든 최소한 상태 판정이 흔들리지 않나?"에 더 가깝다.

---

## DriveState는 지금 어디까지 와 있나
현재 문서 기준으로는 다음 정도까지 이해하면 맞다.

### 이미 되어 있는 것
- `UCFVehicleDriveComp`를 상태 판단 코어로 보는 기준이 확정됨
- `UCFVehicleData` 쪽에 `DriveStateConfig`를 넣어 차량별 설정을 받을 구조가 생김
- `DA_PoliceCar`에 넣을 시작값 가이드가 정리됨
- PIE에서 디버그로 상태 전이를 볼 수 있는 흐름이 있음

### 지금 검증해야 하는 것
- 정지 → 가속 전이가 자연스러운가
- 가속 → 관성 → 제동 전이가 안정적인가
- 후진 진입이 흔들리지 않는가
- 작은 요철에서 너무 쉽게 `Airborne`으로 뜨지 않는가
- 큰 점프/낙하에서 `Airborne`이 정상 진입하는가

### 아직 하지 말아야 하는 것
- 차종별 감각을 너무 일찍 만지는 것
- 코어가 불안정한 상태에서 경찰차 느낌만 먼저 맞추는 것
- 문제 원인이 코어인지 튜닝인지 섞어버리는 것

즉,
**지금은 “멋진 손맛”보다 “안정적인 상태 판정 구조”가 먼저**다.

---

## Wheel Sync는 지금 어떻게 봐야 하나
Wheel Sync는 문서가 제일 많이 분화됐던 영역이라 헷갈리기 쉬운데,
지금은 아주 단순하게 보면 된다.

### 과거
- BP 중심으로 많이 실험했다.
- helper 비교, debug pipe, 단일 축 테스트 같은 과도기 절차가 많았다.
- 그래서 문서도 많이 늘어났다.

### 현재
- 그 과도기 기록은 대부분 Archive로 내려갔다.
- 활성 문서에서는 현재 기준 구조만 남기려는 상태다.
- 핵심은 이제
  **“현재 기준 구조에서 Wheel Sync가 안정적으로 읽히고 검증되느냐”** 다.

즉,
Wheel Sync는 지금 새로운 실험을 늘리는 영역이 아니라,
**현재 구조를 정돈하고 기준선을 문서/검증에 고정하는 영역**으로 보는 게 맞다.

---

## 데이터 설계는 프로젝트에서 어떤 의미를 가지나
현재 문서 중 `Plan/DataPlan/` 쪽을 보면,
이 프로젝트는 단순히 차를 한 대 움직이는 걸 넘어서
**차량 데이터를 정리 가능한 구조로 묶고 싶어 한다**는 것이 보인다.

핵심은 이거다.

- 차량 데이터가 여기저기 흩어져 있으면 관리가 어려움
- 그래서 `UCFVehicleData` 같은 루트 데이터 축으로 정리하려고 함
- 시각, 레이아웃, 움직임, 휠 표현, 레퍼런스 등을 구조적으로 나누려는 방향이 있음

이 말은 곧,
이 프로젝트가 단순 테스트용 차량 하나보다 나중에 **여러 차량 / 여러 세팅 / 여러 차종**으로 확장될 가능성을 염두에 두고 있다는 뜻이다.

다만 여기서 중요한 점은,
**이 데이터 문서군은 아직 전체 게임 기획서가 아니라 차량 데이터 설계 문서군**이라는 점이다.

즉,
현재 문서가 강하게 말해주는 것은
- 차량 코어 설계
- 차량 상태 판단
- 차량 데이터 정리
이지,
게임의 전체 전투 규칙, 모드 구조, progression 전체를 다 말해주지는 않는다.

---

## 그래서 이 프로젝트의 “현재 기획”은 어디까지 확정된 건가
이 부분은 솔직하게 나눠서 봐야 한다.

### 현재 문서로 확실히 말할 수 있는 것
- 이 프로젝트는 차량 플레이를 중심으로 굴러간다.
- 현재 기준 플레이 차량은 `BP_CFVehiclePawn`이다.
- 구조적으로는 BP보다 C++ 코어가 기준이다.
- DriveState와 Wheel Sync가 현재 가장 중요한 코어 영역이다.
- 차량 데이터는 `UCFVehicleData` 중심 구조로 정리하려는 방향이다.
- 테스트 기준 차량 자산은 `DA_PoliceCar`다.

### 현재 문서만으로는 아직 확실히 말하기 어려운 것
- 이 게임의 최종 게임 모드가 정확히 무엇인지
- 전투 규칙이 차량전 중심인지, 다른 시스템과 얼마나 엮이는지
- 맵 구조, 라운드 구조, 승리 조건 같은 상위 게임 기획
- 장기 progression이나 컨텐츠 루프

즉,
**지금 문서군은 “프로젝트 전체 게임 기획서”라기보다 “차량 코어 개발 프로젝트 문서”에 더 가깝다.**

이걸 알면 오히려 덜 답답하다.
문서를 다 읽고도 전체 게임 기획이 안 잡힌 이유는,
내가 놓친 게 아니라 **현재 문서의 무게중심이 거기에 있지 않기 때문**이다.

## 지금 당신이 이 프로젝트를 이해할 때 제일 중요한 문장
이 프로젝트를 한 문장씩 끊어서 머리에 넣으면 이렇게 된다.

### 문장 1
지금 CarFight의 현재 기준 차량은 `BP_CFVehiclePawn`이다.

### 문장 2
실제 규칙과 상태는 `ACFVehiclePawn` + `UCFVehicleDriveComp` + `UCFWheelSyncComp` + `UCFVehicleData` 중심의 C++ 코어에 둔다.

### 문장 3
BP는 여전히 필요하지만, 표현과 조립을 담당하는 Thin BP로 유지한다.

### 문장 4
현재 가장 중요한 개발 과제는 새로운 기능 추가보다,
DriveState 공통 코어와 Wheel Sync 현재 기준을 안정화하는 것이다.

### 문장 5
차종별 손맛 튜닝은 코어 안정화 뒤에 오는 다음 단계다.

### 문장 6
현재 문서군은 전체 게임 기획서보다 차량 코어 개발 문서군에 가깝다.

이 6개만 기억해도,
지금 프로젝트 문서를 읽을 때 길을 훨씬 덜 잃는다.

현재 우선순위를 사람 말로 다시 쓰면 이렇다.

## 마지막 정리
현재 문서 기준으로 CarFight는,
아직 "모든 게임 기획이 한 권으로 완성된 프로젝트"라기보다,
**차량 코어를 C++ 중심으로 안정화하면서, 나중에 더 큰 시스템을 얹을 수 있게 기반을 만드는 단계**에 있다.

그래서 지금 당신이 이 프로젝트를 이해할 때 가장 먼저 잡아야 할 것은
