# CF_VehicleMoveTest

## 문서 목적
본 문서는 `CF_VehicleMoveSpec.md` 기준 차량 입력 해석 변경안의 검증 절차와 합격 기준을 정의한다.

이 문서는 아래를 검증 대상으로 삼는다.

- 각도 기준계가 명세대로 동작하는지
- 쓰로틀 / 후진 / 검은 영역 판정이 맞는지
- 검은 영역 유지 정책이 맞는지
- 전진 중 후진 영역이 브레이크로 해석되는지
- 후진 전환이 Chaos Vehicle `bUseAutoReverse` 기준으로 정상 동작하는지
- 장치 모드별 입력 차단이 맞는지
- 디버그 표시가 충분한지

---

## 1. 테스트 전 준비

### 1-1. 필수 준비물
- 차량 테스트용 Pawn 배치 완료
- 차량 이동용 `Axis2D` 입력 액션 준비
- IMC에 키보드/게임패드 매핑 준비
- Chaos Vehicle에서 `bUseAutoReverse = true` 확인
- PIE에서 현재 차량 속도와 DriveState를 확인할 수 있는 디버그 문자열 준비

### 1-2. 권장 디버그 항목
PIE 중 아래 항목을 화면이나 로그에서 바로 볼 수 있어야 한다.

- RawInputX
- RawInputY
- Magnitude
- AngleDeg
- ResolvedZone
- LastDirectionIntent
- SteeringOutput
- ThrottleOutput
- BrakeOutput
- VehicleSpeedKmh
- ForwardSpeedKmh
- DeviceMode
- bUsedBlackZoneHold
- DriveState

---

## 2. 기본 기능 검증

### TC-01. 각도 기준 검증
**목적**
위/오른쪽/아래/왼쪽 각도 기준이 명세와 일치하는지 확인한다.

**예상 결과**
- 위 = `0°` 근처
- 오른쪽 = `90°` 근처
- 아래 = `180°` 근처
- 왼쪽 = `270°` 근처

### TC-02. 쓰로틀 영역 판정
**목적**
`260° ~ 100°`가 쓰로틀 영역으로 해석되는지 확인한다.

**예상 결과**
- `ResolvedZone = Throttle`
- `Throttle = Magnitude`
- `Brake = 0`
- `LastDirectionIntent = Forward`

### TC-03. 후진 영역 판정
**목적**
`135° ~ 225°`가 후진 영역으로 해석되는지 확인한다.

**예상 결과**
- `ResolvedZone = Reverse`
- 입력 해석 계층은 후진 의도를 유지한다.
- A안 기준 현재 구현에서는 브레이크 전달이 유지된다.
- 실제 후진 전환은 Chaos Vehicle `bUseAutoReverse`가 담당한다.

### TC-04. 검은 영역 판정
**목적**
쓰로틀/후진 외 나머지 각도가 검은 영역으로 해석되는지 확인한다.

**예상 결과**
- `ResolvedZone = Black`
- 조향은 계속 반영된다.
- 최근 방향 의도에 따라 전진 또는 후진 유지 정책이 적용된다.

## 3. 경계각 검증

### TC-05. 시작/종료각 포함 검증
**목적**
경계각이 포함 조건으로 처리되는지 확인한다.

**확인 대상**
- `100°`
- `135°`
- `225°`
- `260°`

**예상 결과**
- `100°` = 쓰로틀 영역 포함
- `135°` = 후진 영역 포함
- `225°` = 후진 영역 포함
- `260°` = 쓰로틀 영역 포함

---

## 4. 검은 영역 유지 정책 검증

### TC-06. 전진 후 검은 영역 진입
**목적**
전진 중 검은 영역 진입 시 전진 의도가 유지되는지 확인한다.

**예상 결과**
- `ResolvedZone = Black`
- `LastDirectionIntent = Forward`
- `Throttle = 현재 Magnitude`
- `Brake = 0`
- `Steering = InputX`
- `bUsedBlackZoneHold = true`

### TC-07. 후진 의도 후 검은 영역 진입
**목적**
후진 의도 상태에서 검은 영역 진입 시 후진 의도가 유지되는지 확인한다.

**예상 결과**
- `ResolvedZone = Black`
- `LastDirectionIntent = Reverse`
- A안 기준 현재 구현에서는 브레이크 유지가 전달된다.
- 조향은 계속 유지된다.

### TC-08. 방향 이력 없음 + 검은 영역
**목적**
방향 이력이 없는 상태에서 검은 영역을 중립으로 해석하는지 확인한다.

**예상 결과**
- `LastDirectionIntent = None`
- `Throttle = 0`
- `Brake = 0`
- `Steering = InputX`

---

## 5. 후진/브레이크 전환 검증

### TC-09. 전진 중 후진 영역 입력 = 브레이크
**목적**
전진 중 후진 영역 입력이 즉시 역추진이 아니라 브레이크로 해석되는지 확인한다.

**예상 결과**
- `Brake = Magnitude`
- `Throttle = 0`
- 즉시 역방향 추진하지 않는다.

### TC-10. `bUseAutoReverse` 기반 후진 전환
**목적**
전진 중 뒤 입력 유지 시 Chaos Vehicle `bUseAutoReverse`가 정지 후 후진 전환을 정상 수행하는지 확인한다.

**절차**
1. 차량을 충분히 전진시킨다.
2. 스틱을 후진 영역에 유지한다.
3. 디버그에서 `Brake`, `Throttle`, `ForwardSpeedKmh`, `DriveState`를 본다.
4. 차량이 감속 후 실제로 뒤로 움직이는지 확인한다.

**예상 결과**
- 감속 구간: `Brake > 0`, `Throttle = 0`
- 정지 후: Chaos Vehicle이 reverse 전환
- 실제 차량이 후진 시작
- 수동 기어 강제나 음수 스로틀 없이도 후진이 동작

### TC-11. 수동 기어 강제 미사용 검증
**목적**
현재 구현이 수동 기어 강제 없이 동작하는지 확인한다.

**예상 결과**
- 후진 동작이 `SetTargetGear(-1)` 같은 강제 로직에 의존하지 않는다.
- `bUseAutoReverse`만으로 후진 전환이 가능하다.

## 6. 조향 검증

### TC-12. 검은 영역 조향 유지
**목적**
검은 영역에서도 조향이 끊기지 않는지 확인한다.

**예상 결과**
- 좌측 입력 시 음수 조향
- 우측 입력 시 양수 조향
- 가감속 해석과 별개로 조향은 계속 반영

## 7. 장치 모드 검증

### TC-13. Auto 모드
- 유효한 입력 액션 값이 들어오면 해석된다.
- 장치 전환을 불필요하게 막지 않는다.

### TC-14. KeyboardMouseOnly 모드
- 키보드/마우스는 동작
- 게임패드는 무시

### TC-15. GamepadOnly 모드
- 게임패드는 동작
- 키보드/마우스는 무시

---

## 8. 키 하드코딩 방지 검증

### TC-16. IMC 리매핑 검증
**목적**
물리 키/스틱 배치를 바꿔도 입력 해석이 유지되는지 확인한다.

**예상 결과**
- 코드 수정 없이 동일한 해석 결과가 나온다.
- 특정 물리 키 이름에 종속되지 않는다.

---

## 9. 회귀 검증

### TC-17. 기존 DriveState 회귀 확인
**목적**
새 입력 해석 계층 도입 후 기존 DriveState 계산이 비정상적으로 깨지지 않았는지 확인한다.

**예상 결과**
- 기존 DriveState 전이 요약 문자열이 여전히 유효하다.
- 극단적으로 상태가 튀지 않는다.

---

## 10. 합격 기준
아래가 모두 만족되면 1차 합격으로 본다.

- 각도 기준계가 명세와 일치한다.
- 경계각이 포함 조건으로 정확히 처리된다.
- 검은 영역이 `HoldDirection_UseCurrentMagnitude`로 동작한다.
- 전진 중 후진 영역은 브레이크로 동작한다.
- `bUseAutoReverse = true` 환경에서 실제 후진 전환이 동작한다.
- 수동 기어 강제 없이 후진이 된다.
- 장치 모드별 차단 규칙이 맞다.
- IMC 리매핑 후에도 코드 수정 없이 동일 의미가 유지된다.
