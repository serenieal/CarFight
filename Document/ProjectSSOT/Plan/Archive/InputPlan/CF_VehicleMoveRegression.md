# CF_VehicleMoveRegression

## 문서 목적
본 문서는 `CF_VehicleMoveSpec.md`, `CF_VehicleMoveTest.md`, `CF_VehicleMoveChecklist.md` 기준으로 **차량 신규 조작 변경 이후의 회귀 검증 범위와 실행 순서**를 정리한 문서다.

이 문서의 목적은 아래 두 가지다.

- 현재 구현이 정상 동작하는 것을 넘어, **기존 기능을 깨지 않았는지** 체계적으로 확인하기
- 테스트 결과를 다음 세션과 다음 작업자도 바로 이어볼 수 있도록 **기록 형식까지 고정**하기

---

## 1. 현재 기준선
현재 구현 기준선은 아래와 같다.

- 게임패드: `IA_VehicleMove` 기반 신규 조작
- 키보드: 기존 `IA_Throttle / IA_Brake / IA_Steering` 기반 조작 유지
- 후진 전환: Chaos Vehicle `bUseAutoReverse`
- 입력 충돌 방지: Runtime Input Ownership (`VehicleMove2D / LegacyAxis / None`)
- IMC 분리 완료
- P1 입력 충돌 방지 구현 완료
- 빌드 성공 및 PIE 실기 테스트 완료

즉, 이번 회귀 검증은 **신규 조작이 잘 되느냐**보다 아래를 더 중점적으로 본다.

- 장치 모드에서 기존 차단 정책이 안 깨졌는가
- IMC 리매핑 후 의미가 유지되는가
- 입력 소유권 전환이 예상치 못한 부작용을 만들지 않는가
- DriveState / 디버그 / 체감이 회귀하지 않았는가

---

## 2. 회귀 검증 우선순위

### Priority 1. 장치 모드 회귀
가장 먼저 확인할 항목이다.

- `Auto`
- `KeyboardMouseOnly`
- `GamepadOnly`

이 항목은 입력 경로 자체가 바뀐 현재 구조에서 가장 먼저 깨질 수 있는 부분이다.

### Priority 2. IMC 리매핑 회귀
두 번째로 확인할 항목이다.

- 게임패드 신규 조작 리매핑
- 키보드 기존 조작 리매핑
- 코드 수정 없이 동일 의미 유지 여부

### Priority 3. Runtime Input Ownership 회귀
세 번째로 확인할 항목이다.

- `VehicleMove2D` -> `LegacyAxis` 전환
- `LegacyAxis` -> `VehicleMove2D` 전환
- release 이후 덮어쓰기 문제 재발 여부
- `InputOwner` 디버그와 실제 체감 일치 여부

### Priority 4. DriveState / 속도 / 후진 체감 회귀
네 번째로 확인할 항목이다.

- DriveState 전이 이상 여부
- 감속/정지/후진 전환 체감
- 검은 영역 유지 정책 체감

---

## 3. 테스트 묶음

## 3-1. 장치 모드 회귀

### RG-01. Auto 모드
**목적**
`Auto` 모드에서 게임패드와 키보드가 각자 의도된 입력 경로로 정상 동작하는지 확인한다.

**절차**
1. `InputDeviceMode = Auto`
2. 게임패드 입력으로 신규 조작 수행
3. 키보드 입력으로 기존 조작 수행
4. `InputOwner`, `DeviceMode`, `DriveState`를 확인한다.

**예상 결과**
- 게임패드 입력 시 신규 조작 해석이 정상 동작
- 키보드 입력 시 기존 축 입력이 정상 동작
- 장치 전환이 과도하게 막히지 않음

### RG-02. KeyboardMouseOnly 모드
**목적**
키보드만 허용되고 게임패드는 차단되는지 확인한다.

**절차**
1. `InputDeviceMode = KeyboardMouseOnly`
2. 키보드 전/후진/조향 입력 확인
3. 게임패드 입력 차단 확인

**예상 결과**
- 키보드는 정상 동작
- 게임패드는 무시됨
- `InputOwner`가 비정상적으로 `VehicleMove2D`로 잠기지 않음

### RG-03. GamepadOnly 모드
**목적**
게임패드만 허용되고 키보드는 차단되는지 확인한다.

**절차**
1. `InputDeviceMode = GamepadOnly`
2. 게임패드 신규 조작 확인
3. 키보드 입력 차단 확인

**예상 결과**
- 게임패드는 정상 동작
- 키보드는 무시됨
- `InputOwner`가 비정상적으로 `LegacyAxis`로 잠기지 않음

---

## 3-2. IMC 리매핑 회귀

### RG-04. 게임패드 신규 조작 리매핑
**목적**
게임패드의 물리 스틱 배치를 바꿔도 `IA_VehicleMove` 의미가 유지되는지 확인한다.

**절차**
1. 게임패드용 IMC에서 차량 이동 입력 매핑을 다른 스틱 또는 다른 축 조합으로 변경
2. PIE 실행
3. 전진/후진/검은 영역/조향이 같은 의미로 동작하는지 확인

**예상 결과**
- 코드 수정 없이 동일한 해석 결과가 나옴
- 특정 스틱 이름에 하드코딩되어 있지 않음

### RG-05. 키보드 기존 조작 리매핑
**목적**
키보드 입력 키를 바꿔도 기존 조작 의미가 유지되는지 확인한다.

**절차**
1. 키보드용 IMC에서 전진/브레이크/조향 키를 변경
2. PIE 실행
3. 기존 조작 의미가 유지되는지 확인

**예상 결과**
- 코드 수정 없이 동일한 의미 유지
- 특정 물리 키 이름에 종속되지 않음

---

## 3-3. Runtime Input Ownership 회귀

### RG-06. VehicleMove2D 우선 적용 확인
**목적**
게임패드 신규 조작과 기존 축 입력이 동시에 살아 있을 때 `VehicleMove2D` 경로가 정상적으로 주도권을 가지는지 확인한다.

**절차**
1. 충돌 상황을 만들 수 있는 IMC로 테스트 환경 구성
2. 게임패드 입력 수행
3. `InputOwner`, 차량 반응, 디버그를 확인

**예상 결과**
- `InputOwner = VehicleMove2D`
- 기존 축 입력이 같은 프레임에 덮어쓰지 않음

### RG-07. LegacyAxis 전환 확인
**목적**
키보드 입력 시작 시 `LegacyAxis`로 자연스럽게 전환되는지 확인한다.

**절차**
1. 게임패드 조작 후 정지 또는 대기
2. 키보드 입력 시작
3. `InputOwner`와 실제 차량 반응 확인

**예상 결과**
- `InputOwner = LegacyAxis`
- 키보드 입력이 정상 적용
- 소유권 전환이 흔들리지 않음

### RG-08. release 경로 회귀
**목적**
release 이벤트가 현재 소유권이 아닌 입력 경로를 0으로 덮어쓰지 않는지 확인한다.

**절차**
1. 게임패드 조작 중 기존 축 release 유발 상황 확인
2. 키보드 조작 중 VehicleMove release 유발 상황 확인
3. 디버그와 차량 반응 확인

**예상 결과**
- release가 다른 입력 경로 값을 망치지 않음
- 이전에 수정한 덮어쓰기 문제가 재발하지 않음

---

## 3-4. DriveState / 후진 / 체감 회귀

### RG-09. DriveState 전이 회귀
**목적**
신규 입력 계층과 입력 소유권 계층 추가 후 DriveState 전이가 비정상적으로 튀지 않는지 확인한다.

**절차**
1. 전진
2. 감속
3. 정지
4. 후진
5. 다시 전진
6. 각 단계의 `DriveState`와 속도 변화를 기록

**예상 결과**
- 상태 전이가 논리적으로 이어짐
- 극단적 튐이나 이상 반복 없음

### RG-10. 검은 영역 유지 체감 회귀
**목적**
검은 영역 유지 정책이 이전보다 이상해지지 않았는지 확인한다.

**절차**
1. 전진 후 검은 영역 진입
2. 후진 의도 후 검은 영역 진입
3. steering 유지 여부와 가감속 체감 확인

**예상 결과**
- 조향은 유지됨
- 전진/후진 의도 유지가 현재 명세와 맞음
- 체감상 급작스럽거나 비정상적인 끊김이 없음

### RG-11. 후진 전환 체감 회귀
**목적**
`bUseAutoReverse` 기반 후진 전환이 입력 충돌 방지 로직 이후에도 이전과 동일하게 동작하는지 확인한다.

**절차**
1. 충분히 전진
2. 뒤 입력 유지
3. 감속 -> 정지 -> 후진 시작까지 확인

**예상 결과**
- 감속 구간은 브레이크
- 정지 후 후진 전환
- 체감상 이전과 비교해 비정상 지연 없음

---

## 4. 실제 실행 순서
회귀 검증은 아래 순서로 도는 것을 권장한다.

1. `RG-01 ~ RG-03` 장치 모드 회귀
2. `RG-06 ~ RG-08` Runtime Input Ownership 회귀
3. `RG-04 ~ RG-05` IMC 리매핑 회귀
4. `RG-09 ~ RG-11` DriveState / 체감 회귀

이 순서를 권장하는 이유는 아래와 같다.

- 먼저 현재 구조의 큰 분기(장치 모드)가 맞는지 확인
- 그다음 가장 최근에 넣은 P1 안정화 로직 확인
- 이후 리매핑 자유도 확인
- 마지막에 체감/상태 전이까지 확인

## 6. 합격 기준
아래를 모두 만족하면 회귀 검증 1차 합격으로 본다.

- 장치 모드별 입력 차단/허용 규칙이 맞다.
- IMC 리매핑 후에도 코드 수정 없이 의미가 유지된다.
- `VehicleMove2D / LegacyAxis / None` 소유권 전환이 안정적이다.
- release 경로가 다른 입력 경로를 덮어쓰지 않는다.
- DriveState 전이가 비정상적으로 튀지 않는다.
- 검은 영역 유지 정책 체감이 현재 명세와 충돌하지 않는다.
- `bUseAutoReverse` 기반 후진 전환이 이전과 동일하거나 더 안정적이다.

---

## 7. 보류 판정 기준
아래 중 하나라도 걸리면 합격이 아니라 **보류**로 둔다.

- 특정 장치 모드에서 입력 차단이 무너짐
- IMC 리매핑 시 코드 종속성이 드러남
- `InputOwner`와 실제 차량 반응이 어긋남
- release 이후 입력이 튀거나 0으로 잘못 덮어써짐
- DriveState가 설명되지 않는 상태 전이를 보임
- 후진 전환 체감이 이전보다 눈에 띄게 나빠짐
