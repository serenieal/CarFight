# CarFight — 01_Roadmap

> 역할: CarFight 프로젝트의 **현재 해야 할 일 / 우선순위 / 완료 조건 / 진행 순서**를 고정한다.  
> 기준 상태 문서: `00_Handover.md`  
> 상위 방향 문서: `03_VisionAlign.md`  
> 마지막 정리(Asia/Seoul): 2026-03-20

---

## 이 로드맵의 성격
이 문서는 아이디어 수집 문서가 아니다.

이 문서의 목적은 아래 4가지다.
- 현재 구현을 기준으로 **지금 무엇부터 해야 하는지** 순서를 고정한다.
- 작업 중간에 새 주제로 새지 않도록 **중지선**을 만든다.
- 현재 차량 코어를 **다차종 확장 가능한 공통 기준선**으로 닫는다.
- 초기 기획의 방향인 **차량 전투 / 물리 상호작용 / 다양한 차량 대응**으로 돌아가기 전에, 무엇을 먼저 끝내야 하는지 분명히 한다.

---

## 이번 개발 사이클의 최상위 목표
이번 사이클의 목표는 아래 한 문장으로 고정한다.

> **현재 구현 기준선을 실측으로 확정하고, 공통 차량 코어를 닫은 뒤, 첫 번째 실제 차량 프로필을 안정화한다.**

즉 이번 사이클은 아래 순서로만 진행한다.
- **현 기준선 실측**
- **공통 코어 검증/정리**
- **첫 차량 프로필 안정화**
- **다음 차량 확장 준비**

이 순서가 끝나기 전에는 차량 전투 규칙, 카메라 연출, HUD, 사운드, 추가 차량 제작으로 넘어가지 않는다.

---

## 현재 구현 기준
### 현재 활성 기준으로 보는 것
- 현재 기준 플레이 차량: `BP_CFVehiclePawn`
- 현재 기준 Native Pawn: `ACFVehiclePawn`
- 현재 기준 주행 코어: `UCFVehicleDriveComp`
- 현재 기준 휠 시각 동기화 코어: `UCFWheelSyncComp`
- 현재 기준 데이터 축: `UCFVehicleData`
- 현재 기준 테스트 자산: `DA_PoliceCar`
- 현재 기준 테스트 맵: `/Game/Maps/TestMap`
- 현재 기준 입력 자산: `IMC_Vehicle_Default`, `IA_Throttle`, `IA_Steering`, `IA_Brake`, `IA_Handbrake`

### 현재 참고만 하고 활성 판단에서 제외하는 것
- `BP_CFVehiclePawn_P0`
  - 구동 가능 시점 저장용 기준선
  - 현재 활성 개발 자산으로 보지 않음

---

## 지금까지 구현 기준으로 확인된 사실
아래는 문서 추정이 아니라, 실제 코드/자산 확인으로 반영된 사실이다.

### 확인된 구조
- `ACFVehiclePawn`가 실제 차량 Pawn 기준 클래스다.
- `UCFVehicleDriveComp`가 Drive 상태와 입력 해석 코어를 담당한다.
- `UCFWheelSyncComp`가 휠 시각 동기화를 담당한다.
- `UCFVehicleData`가 차량 시각 / 레이아웃 / 무브먼트 / 휠 비주얼 / 참조형 자산 / DriveStateConfig를 담는 데이터 축이다.
- `BP_CFVehiclePawn`는 현재 확인된 범위에서 큰 로직 덩어리보다는 얇은 BP 레이어로 사용된다.
- `ApplyVehicleDataConfig()` 흐름상 VehicleData 기반 적용이 실제 코드에 연결되어 있다.
- `ApplyVehicleLayoutConfig()`에는 차체 바운드 기반 자동 휠 위치 적용 흐름이 있다.

### 아직 실측이 더 필요한 부분
- `BP_CFVehiclePawn`의 실제 컴포넌트 구성과 참조 연결 상태
- `DA_PoliceCar`의 실제 세팅값
- `TestMap`에서 현재 어떤 방식으로 기준 차량이 배치/검증되는지
- `UCFWheelSyncComp`의 디버그/과도기 옵션 중 현재 실제 사용 중인 조합

이 4가지는 이번 사이클의 첫 단계에서 반드시 실측으로 닫는다.

## 절대 기준
### 이번 사이클에서 먼저 끝내야 하는 것
1. 현재 기준선 실측
2. DriveState 공통 코어 검증
3. Wheel Sync 현재 운용 방식 정리
4. VehicleData 공통값 / 차량별값 분리 기준 확정
5. `DA_PoliceCar` 첫 번째 차량 프로필 안정화

### 이번 사이클에서 하지 않을 것
- 새 차량을 먼저 실제 플레이어블로 만드는 작업
- 카메라 연출을 먼저 정리하는 작업
- HUD / 사운드 / 시네마틱 반응으로 퍼지는 작업
- 차량 전투 규칙 구현에 먼저 들어가는 작업
- 코어 검증이 끝나기 전에 차량별 손맛을 과도하게 다듬는 작업
- `BP_CFVehiclePawn_P0`를 활성 기준 자산처럼 다시 끌어오는 작업

### 샛길 방지 질문
아래 질문 중 하나라도 "아니오"면 지금 작업을 멈추고 로드맵으로 돌아온다.
- 이 작업이 **현 기준선 실측**에 직접 도움이 되는가?
- 이 작업이 **공통 코어 검증**을 더 명확하게 만드는가?
- 이 작업이 **다차종 확장 준비**를 직접 앞으로 당기는가?

---

## 전체 진행 순서
이번 사이클은 아래 5단계로만 진행한다.

1. **Phase 0 — 현 기준선 실측**
2. **Phase 1 — 공통 코어 폐쇄**
3. **Phase 2 — 첫 차량 프로필 안정화**
4. **Phase 3 — 다음 차량 확장 준비**
5. **Phase 4 — 차량 전투 시스템 진입 준비**

각 Phase의 완료 조건을 만족하기 전에는 다음 Phase로 넘어가지 않는다.

# Phase 0 — 현 기준선 실측
## 목표
현재 프로젝트가 실제로 어떤 자산/값/맵/루틴 위에서 굴러가는지 **말이 아니라 실측 정보**로 확정한다.

이 단계가 끝나야 이후 로드맵이 문서 추정이 아니라 실제 구현 기준이 된다.

## 해야 할 일
### P0-1. `BP_CFVehiclePawn` 실제 구성 확인
확인 항목:
- 어떤 컴포넌트가 붙어 있는가
- `VehicleData`가 어떤 자산에 연결되어 있는가
- Drive / WheelSync / Debug Widget 관련 연결이 어디까지 BP에서 드러나는가
- 컴포넌트 이름이 코드의 기대 이름과 맞는가
  - `SM_Chassis`
  - `Wheel_Mesh_FL`
  - `Wheel_Mesh_FR`
  - `Wheel_Mesh_RL`
  - `Wheel_Mesh_RR`

기록 방식:
- 실제 구성표를 문서로 남긴다.
- BP에서만 존재하는 연결과 C++가 기대하는 연결을 구분해서 적는다.

### P0-2. `DA_PoliceCar` 실제 값 확인
확인 항목:
- `VehicleVisualConfig`
- `VehicleLayoutConfig`
- `VehicleMovementConfig`
- `WheelVisualConfig`
- `VehicleReferenceConfig`
- `DriveStateConfig`

기록 방식:
- 현재 값을 가능한 그대로 정리한다.
- 기본값인지 override인지 구분해서 적는다.
- 이후 튜닝 대상인지, 공통 기본값 후보인지 표시한다.

### P0-3. `TestMap` 현재 검증 루프 확인
확인 항목:
- 기준 차량이 어떻게 배치되는가
- 플레이 시작 후 기본 조작 루프가 어떻게 이어지는가
- 디버그 확인이 가능한가
- 현재 맵이 공통 코어 검증용으로 충분한가

기록 방식:
- TestMap에서 어떤 순서로 검증하는지 절차를 적는다.
- 이후 반복 테스트 루틴의 기본형으로 삼는다.

### P0-4. WheelSync 현재 운용 방식 확인
확인 항목:
- 실제 사용 중인 옵션 조합
- 디버그 파이프가 현재 기준인지, 보조 경로인지
- helper compare / legacy 옵션이 실제 사용 중인지, 단순 잔존 필드인지
- 현재 검증에서 반드시 켜야 하는 옵션과 꺼야 하는 옵션

기록 방식:
- **현재 기준 설정**과 **현재 미사용/정리 대상**을 분리한다.

### P0-5. DriveState 디버그 확인
확인 항목:
- `bEnableDriveStateOnScreenDebug`
- `bShowDriveStateTransitionSummary`
- 상태 문자열 가독성
- 현재 디버그가 코어 문제 추적에 충분한가

### P0-6. 활성 기준 문서 보강
- `00_Handover.md`에 실측 기준을 반영한다.
- `08_P0_Verification.md`의 기준 항목을 실측 기준으로 조정한다.
- 필요하면 별도 실측 메모를 새로 만든다.

## 완료 조건
- [ ] `BP_CFVehiclePawn` 실제 구성표가 있다.
- [ ] `DA_PoliceCar` 실제 설정표가 있다.
- [ ] `TestMap` 기준 검증 루틴이 적혀 있다.
- [ ] WheelSync 현재 기준 설정과 미사용/정리 대상을 분리했다.
- [ ] 이후 문서가 실측 정보를 기준으로 설명 가능하다.

## 산출물
- 실측 메모
- 업데이트된 `00_Handover.md`
- 업데이트된 `08_P0_Verification.md`
- 필요 시 새 기준표 문서

## 이 Phase에서 금지
- 튜닝부터 먼저 시작하기
- 새로운 차량 자산 제작 시작하기
- 전투 규칙 설계로 주제를 옮기기
- 실측 없이 문서 문장부터 더 예쁘게 다듬기

# Phase 1 — 공통 코어 폐쇄
## 목표
현재 코어가 특정 테스트 차량 하나가 아니라 **다른 차량에도 옮겨갈 수 있는 공통 구조**인지 확인하고, 아니면 그 기준으로 정리한다.

## 해야 할 일
### P1-1. DriveState 공통 코어 검증
`19_DriveState_CoreChecklist.md` 기준으로 아래 흐름을 검증한다.
- 정지 → 전진 진입
- 전진 유지
- 입력 해제 후 Coasting
- 브레이크 진입
- Idle 복귀
- 후진 진입
- 전진↔후진 경계 안정성
- 작은 요철 Grounded 유지
- 큰 점프/낙하 Airborne 진입
- 착지 후 상태 복귀

### P1-2. 코어 문제와 튜닝 문제 분리
실패 항목마다 아래로 분리한다.
- 공통 코어 문제
- 현재 차량 수치 문제
- 디버그 부족 문제
- 맵/검증 환경 문제

### P1-3. WheelSync 공통 규칙 정리
정리 내용:
- 현재 기준으로 남길 설정
- 디버그용으로만 남길 설정
- 문서에서 현재 기준처럼 보이면 안 되는 설정
- 나중에 코드/문서 정리 후보

### P1-4. VehicleData 분리 기준 정리
구분 항목:
- 공통 기본값으로 남길 것
- 차량별 override로 열 것
- 아직 열지 않고 보류할 것

예시 축:
- DriveStateConfig
- WheelVisualConfig
- VehicleMovementConfig
- ReferenceConfig

### P1-5. 현재 결정 로그 보강
`16_CPP_DecisionLog.md`에 아래를 명시한다.
- 공통 기준선
- 차량별 차이 허용 범위
- `DA_PoliceCar`의 위치
- WheelSync 현재 판단

## 완료 조건
- [ ] DriveState 핵심 항목 대부분이 통과다.
- [ ] 실패 항목이 코어 문제인지 튜닝 문제인지 분리되어 있다.
- [ ] WheelSync 현재 기준 설정이 문서에 정리되어 있다.
- [ ] VehicleData 공통값 / 차량별값 구분표가 있다.
- [ ] 현재 구조를 다른 차량에 옮길 수 있는지 설명 가능하다.

## 산출물
- 업데이트된 `19_DriveState_CoreChecklist.md`
- 업데이트된 `08_P0_Verification.md`
- 업데이트된 `16_CPP_DecisionLog.md`
- VehicleData 분리표

## 이 Phase에서 금지
- 경찰차 손맛 완성부터 먼저 파고들기
- 카메라나 HUD 작업으로 이동하기
- 실패 원인을 느낌으로만 적고 넘어가기
- WheelSync 과도기 옵션을 이유 없이 늘리기

# Phase 2 — 첫 차량 프로필 안정화
## 목표
`DA_PoliceCar`를 단순 테스트 자산이 아니라 **첫 번째 실제 차량 프로필**로 안정화한다.

## 해야 할 일
### P2-1. 시작값 입력
`18_DriveState_Tuning.md` 기준으로 현재 시작값을 정리한다.

### P2-2. 조정 순서 고정
아래 순서로만 조정한다.
1. `Idle`
2. `Reversing`
3. `Airborne`

### P2-3. 변경 기록 남기기
각 변경마다 아래를 기록한다.
- 변경 항목
- 이전 값
- 새 값
- 확인한 현상
- 결과

### P2-4. 플레이 감각 최소 기준 정의
이 단계에서의 목표는 “완벽한 손맛”이 아니라 아래다.
- 기본 조작이 읽힌다.
- 상태 전이가 납득된다.
- 작은 요철과 큰 점프의 구분이 된다.
- 후진 진입이 과도하게 답답하지 않다.

## 완료 조건
- [ ] `DA_PoliceCar`의 현재 시작값이 문서화되어 있다.
- [ ] 조정 순서와 변경 이유가 남아 있다.
- [ ] 공통 코어 문제를 수치로 덮지 않았다.
- [ ] 경찰차가 현재 플레이 가능한 첫 프로필로 설명 가능하다.

## 산출물
- 업데이트된 `18_DriveState_Tuning.md`
- 차량별 튜닝 메모

## 이 Phase에서 금지
- 경찰차를 최종 대표 차량처럼 확정 짓기
- 아직 없는 다른 차량 감각까지 상상으로 맞추기
- 손맛 욕심 때문에 공통 기준을 무너뜨리기

# Phase 3 — 다음 차량 확장 준비
## 목표
두 번째 차량을 붙일 수 있도록 **확장 절차와 데이터 규칙**을 닫는다.

## 해야 할 일
### P3-1. 두 번째 차량 후보 선정
현재 자산 후보군 예시:
- SportsCar 계열
- OffroadCar 계열

선정 기준:
- 현재 구조와 차이가 분명할 것
- VehicleData 분리 효과를 확인하기 좋을 것
- 너무 많은 신규 제작 없이 붙일 수 있을 것

### P3-2. 신규 VehicleData 생성 절차 문서화
절차 예시:
1. 새 VehicleData 생성
2. Visual/Layout/Reference 연결
3. 공통 기본값 적용
4. 차량별 override 최소 개방
5. TestMap 반복 검증
6. 차량별 튜닝 진입

### P3-3. 공통값 / 차량별값 최종 분리표 보강
이 단계에서 분리표를 실제 신규 차량 적용 전 기준 문서로 승격한다.

### P3-4. 반복 검증 루틴 정리
- 어떤 순서로 맵에서 테스트할지
- 어떤 문서에 결과를 남길지
- 무엇이 통과면 다음 단계로 가는지

## 완료 조건
- [ ] 두 번째 차량 후보가 선정되어 있다.
- [ ] 신규 VehicleData 생성 절차가 적혀 있다.
- [ ] 공통값 / 차량별값 분리 기준이 실사용 가능 수준이다.
- [ ] 반복 검증 루틴이 정리되어 있다.

## 산출물
- 확장 메모
- VehicleData 분리표 보강본
- 다음 차량 추가 절차서

## 이 Phase에서 금지
- 후보만 여러 개 늘리고 실제 절차를 안 적는 것
- 신규 차량 제작 작업으로 바로 뛰어드는 것
- 공통 기준이 안 닫힌 상태에서 차량 수만 늘리는 것
