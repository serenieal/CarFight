이 경로 이하에서는 언리얼 C++ 코드 작업 규칙을 따른다.

코드를 작성할 때는 다음 규칙을 반드시 지킨다.
- 모든 변수 바로 위에 해당 변수의 역할을 설명하는 한 줄 주석을 작성한다.
- 모든 함수 바로 위에 해당 함수의 역할을 설명하는 한 줄 주석을 작성한다.
- 변수명, 함수명, 파일명, 클래스명은 직관적이고 의미가 분명하게 짓는다.
- 기존에 제안한 코드가 있다면 그 스타일과 구조를 최대한 유지한다.
- 코드를 수정하거나 개선할 때는 반드시 버전 표기(v1.0, v1.1 등)를 한다.
- 코드 변경 시 변경점(Changelog)과 마이그레이션 지침을 함께 제공한다.
- 기존 코드의 함수 시그니처, 파일 경로, 구조를 말 없이 바꾸지 않는다.
- 각 코드 블록을 보여주기 전에 정확한 파일 경로와 파일명, 작업 유형(신규 / 수정 / 교체 / 추가)을 먼저 적는다.
- 새로 제안하는 파일명과 클래스명은 32자를 넘기지 않는다.

Public/Private 경로 규칙을 준수한다.
무분별한 리네이밍을 금지한다.

언리얼 C++에서는 다음을 항상 확인한다.
- 라이프사이클 영향
- 초기화 순서
- Authority / Client 차이
- Replication 영향
- Blueprint 노출 영향
- 에셋 참조 영향

## 이성수준 기준
- 낮음: 파일 탐색, 참조 추적, 좁은 범위 설명, 간단한 보일러플레이트 작성
- 중간: 일반 기능 구현, 소규모 수정, 테스트 보강, 소규모 리팩토링
- 높음: 디버깅, 리뷰, 다중 파일 변경, 구조 변경, Replication / Authority 영향 분석
- 간단한 작업에 불필요하게 높은 이성수준을 사용하지 않는다.
- 다중 파일 영향이나 상태 전이 검토가 필요하면 보수적으로 높음으로 올린다.

문제 해결 시:
- 추측하지 말고 불확실성을 분리한다.
- 가장 안전한 기본안을 먼저 제시한다.
- 구현 후 검증 방법, 체크포인트, 예상 결과를 함께 적는다.
