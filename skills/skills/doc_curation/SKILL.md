# 목적

기존 문서를 더 길게 쓰는 것이 아니라 더 유용하게 정제한다.

핵심 요구사항, 제약사항, 결정 이유, 예외 조건, 검증 기준, 작업 맥락은 유지하고,
중복되거나 가치가 낮은 표현만 줄인다.

목표는 다음 세 가지를 동시에 만족하는 것이다.

- 정보 손실 없이 정보 밀도를 높인다
- 가독성과 구조를 개선한다
- 전체 재작성 대신 최소 수정으로 문서를 더 실용적으로 만든다

---

# 언제 사용할지

다음과 같은 경우에 사용한다.

- 문서 정리, 명세 정제, 설계서 압축, 중복 제거, 구조 개선 요청이 들어왔을 때
- 이미 존재하는 문서를 유지보수해야 하고, 전체 재작성보다 최소 수정이 더 적절할 때
- 같은 내용을 여러 군데에서 반복하거나, 설명은 길지만 실제 의사결정에 필요한 정보는 적을 때
- 요구사항, 제약사항, 예외, 검증 기준을 보존한 채 문서를 더 짧고 명확하게 만들어야 할 때
- 초보자도 이해해야 해서 배경 설명은 남기되, 장황한 반복 설명은 걷어내야 할 때
- 합의된 문서의 의미는 유지하면서 표현과 구조만 정리해야 할 때

예시 요청:

- 이 문서 중복 좀 정리해줘
- 이 스펙 더 짧게 만들되 요구사항은 다 남겨줘
- 설계서 군더더기 걷어내고 구조 정리해줘
- 같은 말 반복되는 부분 합쳐줘
- 문서 전체 다시 쓰지 말고 필요한 부분만 정리해줘
- rationale은 남기고 더 간결하게 만들어줘

---

# 언제 사용하지 않을지

다음 경우에는 기본안으로 사용하지 않는다.

- 문서가 거의 없어서 정제보다 신규 작성이 필요한 경우
- 원문 자체가 부정확하거나 핵심 정보가 누락되어 먼저 사실 확인이 필요한 경우
- 법률, 계약, 규정처럼 표현 하나의 법적 의미가 중요한 문서를 임의 축약하는 작업
- 사용자가 전체 재작성, 새로운 톤앤매너 적용, 대규모 구조 개편을 명시적으로 요청한 경우
- 삭제 허용 범위가 불명확하고 손실 위험이 큰 경우
- 서로 충돌하는 정보가 많은데 어떤 내용이 최신인지 확인되지 않은 경우
  - 이때는 정리보다 충돌 표기와 확인 필요 항목 정리가 우선이다

---

# 핵심 편집 원칙

- 길이보다 정보 밀도와 유용성을 우선한다.
- 문서 전체를 갈아엎지 말고 가능한 한 최소 범위만 수정한다.
- 핵심 요구사항, 제약사항, 결정 이유, 예외 조건, 검증 기준, 작업 맥락은 유지한다.
- 같은 의미의 중복 설명은 하나의 더 강한 문장이나 더 짧은 단락으로 통합한다.
- 장황한 설명은 더 짧고 명확하게 바꾸되, 초보자가 놓칠 배경 설명은 남긴다.
- 설명을 줄일 때는 “무엇을 해야 하는지”, “왜 그렇게 결정되었는지”, “언제 예외가 생기는지”가 사라지지 않게 한다.
- 문서의 기존 의도, 용어, 합의된 표현을 불필요하게 바꾸지 않는다.
- 삭제나 축약이 발생하면 이유와 영향을 `Change Note`에 남긴다.
- 삭제보다 통합과 축약을 우선한다.
- 구조 개선은 허용하지만, 기존 제목 구조가 충분히 기능하면 유지한다.
- 구조 변경은 중복이나 혼선 해소에 필요한 최소 범위에서만 수행한다.

---

# 충돌 내용 처리 원칙

서로 충돌하는 요구사항, 설명, 예외 조건, 검증 기준이 있으면 임의로 한쪽을 조용히 삭제하지 않는다.

다음 우선순위로 처리한다.

1. 문맥상 최신 또는 합의된 내용이 명확하면 그 기준으로 정리한다.
2. 확실하지 않으면 충돌 상태를 드러낸다.
3. 필요하면 아래 표기를 사용한다.
   - `Open Question`
   - `Note`
   - `Deprecated`
   - `Superseded`
4. 충돌을 해소하지 못한 항목은 `Change Note` 또는 별도 확인 필요 항목에 남긴다.

중요:
- 깔끔해 보이게 만들기 위해 근거 없이 한쪽 내용을 제거하지 않는다.
- 충돌 정리는 허용되지만, 의미 왜곡은 허용하지 않는다.

---

# 정제 우선순위

1. 사실상 같은 내용을 반복하는 문장, 단락, 예시를 먼저 통합한다.
2. 결론 없이 길기만 한 설명을 줄이고 핵심 문장으로 재작성한다.
3. 산재한 요구사항, 제약사항, 예외 조건, 검증 기준을 가까운 위치로 재배치한다.
4. 독자가 작업 판단에 직접 쓰지 않는 장식성 문장, 과도한 서론, 자기반복 문구를 줄인다.
5. 필요하면 제목과 목록 구조만 최소 조정해 읽는 흐름을 개선한다.
6. 정보 손실 위험이 있는 삭제는 마지막에 검토하고, 가능하면 삭제보다 통합과 축약을 우선한다.
7. 충돌하는 정보가 있으면 삭제보다 상태 표기와 확인 필요 항목 정리를 우선한다.

---

# 작업 절차

1. 문서의 목적과 독자를 먼저 파악한다.
2. 수정 대상 문서에서 다음 정보를 표시한다.
   - 핵심 요구사항
   - 제약사항
   - 결정 이유
   - 예외 조건
   - 검증 기준
   - 작업 맥락
3. 같은 뜻을 반복하는 구간과 길지만 효용이 낮은 구간을 분리한다.
4. 서로 충돌하거나 최신 여부가 불분명한 구간이 있는지 확인한다.
5. `references/checklist.md`를 기준으로 삭제 가능 영역과 반드시 보존할 영역을 빠르게 점검한다.
6. 전체 재구성 대신 가장 작은 편집 단위부터 수정한다.
7. 여러 곳에 흩어진 동일 정보를 하나로 통합하고, 남은 위치에는 참조 또는 더 짧은 표현만 둔다.
8. 장황한 설명을 줄일 때는 초보자가 이해할 최소 배경 설명이 남는지 확인한다.
9. 구조 변경이 필요하면 제목/섹션 이동은 최소 범위에서만 수행한다.
10. 삭제, 축약, 통합, 충돌 표기가 있었다면 `Change Note`를 작성한다.
11. 최종본에서 요구사항 누락, 예외 누락, 검증 기준 누락, 결정 이유 누락이 없는지 다시 확인한다.
12. 사용자가 원하면 `references/doc_tpl.md` 형식으로 정제 결과를 정리해 제공한다.

---

# 금지 사항

- 정보 확인 없이 의미를 바꿔 쓰지 않는다.
- 요구사항, 제약사항, 예외 조건, 검증 기준, 결정 이유를 조용히 삭제하지 않는다.
- 문서 전체를 새로 쓰는 방식으로 불필요하게 범위를 키우지 않는다.
- 분량을 줄이기 위해 배경 설명을 전부 제거하지 않는다.
- 같은 의미인지 확실하지 않은 문장을 임의로 합치지 않는다.
- 서로 충돌하는 내용을 근거 없이 한쪽만 남기고 정리하지 않는다.
- 삭제나 축약이 있었는데 `Change Note`를 생략하지 않는다.
- 문서의 합의된 용어를 근거 없이 다른 용어로 치환하지 않는다.
- 구조를 보기 좋게 바꾸려다 문서의 원래 의도나 책임 경계를 흐리지 않는다.

# 출력 형식 가이드

- 먼저 `정제 결과`로 무엇을 유지했고 무엇을 줄였는지 짧게 요약한다.
- 이어서 `변경된 문서` 또는 `권장 수정안`을 제시한다.
- 삭제, 축약, 통합, 충돌 표기가 있었다면 반드시 `Change Note`를 붙인다.
- 문서가 길면 수정된 구간만 제시하고, 원문 전체 재출력은 필요할 때만 한다.
- 가능하면 아래 순서를 유지한다.

## 권장 출력 순서

1. 대상 문서와 정제 목표
2. 유지한 핵심 정보
3. 정제된 문서 또는 수정 구간
4. Change Note
5. 빠른 검증 결과

# Change Note 템플릿

아래 형식을 필요에 맞게 채운다.

```md
## Change Note
- 변경 유형:
  - 정리 / 축약 / 통합 / 구조 개선 / 충돌 표기
- 유지한 핵심:
  - 요구사항:
  - 제약사항:
  - 결정 이유:
  - 예외 조건:
  - 검증 기준:
  - 작업 맥락:
- 축약 또는 삭제:
  - 항목:
  - 이유:
  - 정보 손실 여부: 없음 / 검토 필요
- 통합한 중복:
  - 항목:
  - 통합 방식:
- 충돌 또는 확인 필요:
  - 항목:
  - 상태: Open Question / Deprecated / Superseded / Note
