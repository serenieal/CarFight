## 현재 기준 메모
- 현재 기준 플레이 차량: `BP_CFVehiclePawn`
- 현재 기준 Native Pawn: `ACFVehiclePawn`
- 현재 기준 주행 코어: `UCFVehicleDriveComp`
- 현재 기준 휠 시각 동기화 코어: `UCFWheelSyncComp`
- 현재 기준 데이터 축: `UCFVehicleData`
- 현재 기준 테스트 자산: `DA_PoliceCar`

---

## 현재 기준 결정
### CF-DL-0021 — 현재 구현 조합을 공통 기준선으로 유지
- 판정: 확정
- 결정:
  - 현재 기준 구현은 `ACFVehiclePawn` / `BP_CFVehiclePawn` / `UCFVehicleDriveComp` / `UCFWheelSyncComp` / `UCFVehicleData` 조합으로 유지한다.
- 이유:
  - 실제 구현 기준선과 문서 기준선을 일치시켜야 이후 차량 확장이 쉬워진다.

---

### CF-DL-0022 — DriveComp는 공통 상태 판단 코어
- 판정: 확정
- 결정:
  - `UCFVehicleDriveComp`는 단순 입력 전달자가 아니라 공통 상태 판단 코어로 둔다.
- 이유:
  - 차량별 차이를 열기 전에 공통 규칙을 한곳에 모아야 한다.

---

### CF-DL-0023 — BP_CFVehiclePawn은 Thin BP 유지
- 판정: 확정
- 결정:
  - `BP_CFVehiclePawn`는 표현 / 조립 / 반응 중심 Thin BP로 유지한다.
  - 새 상태 판정 / 복잡 분기 / 중복 규칙을 추가하지 않는다.
- 이유:
  - 차량별 확장 시에도 책임 분리를 유지하기 쉽다.

---

### CF-DL-0024 — WheelSync는 공통 규칙 우선
- 판정: 확정
- 결정:
  - `UCFWheelSyncComp`는 특정 상황 전용 분기보다 공통 차량 구조에서 재사용 가능한 기준을 우선한다.
- 이유:
  - 다차종 확장 시 특정 테스트 상황용 규칙이 많아지면 유지가 어려워진다.

---

### CF-DL-0025 — VehicleData는 공통값과 차량별값을 분리하는 축
- 판정: 확정
- 결정:
  - `UCFVehicleData`는 공통 코어와 차량별 차이를 연결하는 데이터 축으로 둔다.
  - 차량별 차이는 가능하면 VehicleData에서 여는 방향을 우선 검토한다.
- 이유:
  - 코드 복붙 없이 차종별 확장을 하려면 데이터 분리가 필요하다.

---

### CF-DL-0026 — 테스트 차량은 기준선이지 종착점이 아니다
- 판정: 확정
- 결정:
  - `DA_PoliceCar`는 현재 첫 기준 차량이며, 최종 목표 차량처럼 취급하지 않는다.
- 이유:
  - 공통 코어가 특정 차량만을 위한 구조로 굳는 것을 막아야 한다.

---

### CF-DL-0027 — 공통 코어 검증 후 차량별 튜닝
- 판정: 확정
- 결정:
  - 차종별 감각 조정보다 공통 코어 검증을 먼저 수행한다.
- 운영 원칙:
  - 코어 검증 시에는 기본값 기준으로 본다.
  - 세부 차종 튜닝은 공통 코어 통과 뒤 진행한다.

---

### CF-DL-0028 — 현재 단계는 확장 준비 단계
- 판정: 확정
- 결정:
  - 현재 문서와 작업의 중심 표현은 단순 유지가 아니라 **공통 기준선 정리와 확장 준비**로 둔다.
- 이유:
  - 지금의 목표는 다음 차량과 차량 전투 시스템으로 이어질 수 있는 기반을 만드는 것이기 때문이다.
