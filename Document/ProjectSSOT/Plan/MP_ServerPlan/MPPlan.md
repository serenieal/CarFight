# CarFight Dedicated Server 현재 구현 기능 안정화 계획서

- 문서 버전: 0.2.0
- 작성일: 2026-05-22
- 대상 프로젝트: CarFight
- 저장 위치: `Document/ProjectSSOT/Plan/MP_ServerPlan/MPPlan.md`
- 현재 범위 기준 문서: `Scope.md`

---

## 1. 계획 목적

CarFight는 초반에 멀티플레이를 고려하지 않고 개발을 진행했지만, 이후 Dedicated Server 기반 멀티플레이를 목표로 전환한다.

다만 현재 단계에서는 신규 게임플레이 시스템을 만들지 않는다.

현재 계획의 목적은 다음과 같다.

```text
이미 구현되어 있는 기능만 Dedicated Server 기준으로 점검하고 안정화한다.
```

즉, 현재 작업은 무기 시스템, 체력 시스템, 스코어 시스템 같은 확장 시스템을 만드는 단계가 아니다.

---

## 2. 현재 최종 결정

### 2.1 현재 작업 범위는 구현된 기능으로 제한한다

현재 단계의 작업 범위는 다음과 같이 제한한다.

```text
- 현재 프로젝트에 이미 구현되어 있는 기능만 확인한다.
- 현재 프로젝트에 이미 구현되어 있는 기능만 수정한다.
- 현재 프로젝트에 이미 구현되어 있는 기능만 Dedicated Server에서 안정화한다.
```

실제 프로젝트에 아직 없는 기능은 현재 작업 범위에서 제외한다.

---

### 2.2 새 시스템을 먼저 만들지 않는다

아래 기능은 현재 단계에서 새로 만들지 않는다.

```text
- 무기 시스템
- 체력 시스템
- 대미지 시스템
- 스코어 시스템
- 킬/데스 시스템
- 리스폰 시스템
- 킬 로그
- 스코어보드
- 매치 승패
- 로비
- Steam 세션
- 매치메이킹
- 서버 브라우저
- 서버 배포 자동화
```

이 기능들은 삭제되거나 포기된 것이 아니다.

현재 구현 기능 안정화가 끝난 뒤 확장 단계에서 다시 계획한다.

---

### 2.3 Dedicated Server 기준은 지금부터 적용한다

새 시스템을 만들지는 않지만, 기존 기능을 점검할 때는 Dedicated Server 기준을 적용한다.

기본 원칙은 다음과 같다.

```text
클라이언트:
- 입력 처리
- 카메라
- HUD
- 사운드
- 이펙트
- 소유자 전용 피드백

서버:
- 실제 게임 상태
- Actor Spawn
- Pawn 소유권
- 복제 기준 상태
- 현재 구현된 기능 중 서버 권한이 필요한 처리
```

Dedicated Server에는 로컬 플레이어, 화면, HUD, 카메라, 로컬 입력이 없다는 전제로 작업한다.

## 3. 현재 포함 범위

아래 기능은 실제 프로젝트에 이미 구현되어 있는 경우에만 현재 안정화 대상으로 삼는다.

```text
- 차량 이동
- 차량 입력
- 차량 소유권
- 차량 카메라
- HUD / UI
- 부스터
- 드리프트
- 차량 충돌
- 차량 스폰
- 차량 선택
- 사운드
- 이펙트
- 맵 오브젝트
- 디버그 기능
```

중요:

```text
목록에 있더라도 실제 프로젝트에 구현되어 있지 않으면 작업하지 않는다.
```

현재 작업에서 제외하는 항목은 다음과 같다.

```text

- 투사체
- 탄약
- 장전
- 체력
- 대미지
- 사망

```

이 항목들은 문서에 후보로 언급될 수는 있지만, 현재 작업 완료 기준에는 포함하지 않는다.

---

## 5. 기존 기능 점검 기준

현재 구현된 기능은 아래 질문으로 점검한다.

```text
1. 이 기능이 실제로 현재 구현되어 있는가?
2. 이 기능은 서버에서 실행되어야 하는가?
3. 이 기능은 클라이언트에서만 실행되어야 하는가?
4. 모든 플레이어에게 보여야 하는가?
5. 소유자에게만 보여야 하는가?
6. Dedicated Server에서 실행되면 안 되는 코드가 있는가?
7. 클라이언트가 실제 게임 상태를 직접 확정하고 있지 않은가?
8. Actor Spawn이 서버에서 이루어지는가?
9. 복제되어야 하는 Actor에 bReplicates가 설정되어 있는가?
10. UI, 카메라, 사운드 코드가 서버 로직에 섞여 있지 않은가?
```

## 6. 위험 코드 유형

기존 코드나 Blueprint에서 아래 패턴이 보이면 현재 안정화 단계에서 우선 점검한다.

```text
- GetPlayerController(0)
- GetFirstPlayerController()
- Pawn 내부에서 CreateWidget 직접 호출
- GameMode 또는 서버 흐름에서 UI 생성
- 서버 코드에서 카메라 컴포넌트 접근
- 서버 코드에서 로컬 사운드 재생 기대
- 입력 함수에서 실제 상태를 바로 변경
- 클라이언트에서 게임플레이 Actor를 Spawn하고 끝냄
- 이펙트와 실제 상태 변경이 한 함수에 섞여 있음
- 복제 대상 Actor인데 bReplicates가 꺼져 있음
- GameMode 값을 클라이언트 UI에서 직접 읽으려 함
```

## 7. 현재 단계별 작업 계획

### Phase 1. 실제 구현 기능 목록 작성

목표:

```text
프로젝트에 현재 실제로 존재하는 기능만 목록화한다.
```

작업:

```text
1. C++ 클래스 목록 확인
2. Blueprint Asset 목록 확인
3. 현재 GameMode 확인
4. 현재 PlayerController 확인
5. 현재 Pawn 또는 Vehicle Pawn 확인
6. 현재 UI 생성 방식 확인
7. 현재 차량 관련 기능 확인
```

산출물:

```text
FeatureList.md 갱신
```

완료 기준:

```text
실제 구현된 기능만 FeatureList.md에 기록되어 있다.
```

---

### Phase 2. 현재 구현 기능 네트워크 점검

목표:

```text
현재 구현된 기능 중 Dedicated Server에서 문제가 될 가능성이 있는 부분을 찾는다.
```

작업:

```text
1. UI 생성 위치 확인
2. 카메라 접근 위치 확인
3. 사운드/이펙트 호출 위치 확인
4. Actor Spawn 위치 확인
5. Pawn 소유권 확인
6. 입력 처리 위치 확인
7. 복제 설정 확인
```

산출물:

```text
NetAudit.md 갱신
```

완료 기준:

```text
현재 구현된 기능의 위험 요소가 기록되어 있다.
```

---

### Phase 3. Dedicated Server 최소 실행선 확인

목표:

```text
Dedicated Server에서 현재 구현된 플레이 흐름이 최소한 깨지지 않는지 확인한다.
```

작업:

```text
1. Dedicated Server 실행
2. 클라이언트 2개 접속
3. 각 클라이언트가 자기 Pawn 또는 차량을 소유
4. 자기 입력이 자기 Pawn 또는 차량에만 적용
5. 현재 복제되어야 하는 이동/상태가 다른 클라이언트에서 보임
6. 서버 로그에 UI/카메라/로컬 사운드 관련 오류 없음
```

산출물:

```text
DSMinRun.md 갱신
```

완료 기준:

```text
DSMinRun.md의 필수 완료 기준을 통과한다.
```

---

### Phase 4. 현재 구현 기능 안정화

목표:

```text
Phase 2와 Phase 3에서 발견된 문제 중 현재 구현 기능에 해당하는 것만 수정한다.
```

작업 원칙:

```text
- 미구현 시스템을 새로 만들지 않는다.
- 현재 구현된 기능의 오류만 수정한다.
- 서버 전용 로직과 클라이언트 전용 연출을 분리한다.
- Dedicated Server에서 의미 없는 HUD, 카메라, 사운드 의존을 제거한다.
```

완료 기준:

```text
현재 구현된 기능이 Dedicated Server 최소 실행선에서 안정적으로 동작한다.
```

---

## 8. 현재 단계 완료 기준

현재 안정화 단계는 아래 조건을 만족하면 완료로 본다.

```text
1. 현재 구현된 기능 목록이 정리되어 있다.
2. 현재 구현된 기능의 서버/클라이언트 책임이 분류되어 있다.
3. Dedicated Server에서 클라이언트 2개가 접속할 수 있다.
4. 각 클라이언트가 자기 Pawn 또는 차량을 올바르게 소유한다.
5. 현재 구현된 입력/이동 기능이 소유자 기준으로 동작한다.
6. 다른 클라이언트에서 보여야 하는 현재 구현 기능이 정상적으로 보인다.
7. Dedicated Server에서 HUD, 카메라, 로컬 사운드 때문에 오류가 나지 않는다.
8. 아직 구현되지 않은 기능을 새로 만들지 않고 안정화 작업을 마친다.
```

---

## 9. 확장 단계 진입 조건

아래 조건을 만족한 뒤에만 무기, 체력, 스코어 같은 확장 시스템을 다시 논의한다.

```text
1. 현재 구현 기능 안정화 완료
2. Dedicated Server 최소 실행선 통과
3. 현재 구현 기능의 위험 코드 목록 정리 완료
4. 서버/클라이언트 책임 분리 완료 또는 보류 사유 기록
5. 다음 확장 대상 하나를 선택
```

확장 단계에서는 한 번에 여러 시스템을 만들지 않는다.
