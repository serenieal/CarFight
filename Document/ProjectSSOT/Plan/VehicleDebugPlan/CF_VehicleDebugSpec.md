# VehicleDebug v2 요구 정리

- Version: 0.1.0
- Date: 2026-04-22
- Status: Draft
- Scope: VehicleDebug 확장 목표, 범위, 비범위, 품질 기준

---

## 1. 문서 목적

이 문서는 현재 `VehicleDebug` 기능을 확장할 때, 무엇을 만들고 무엇을 나중으로 미룰지 먼저 고정하기 위한 요구 정리 문서다.

현재 상태 문서 기준으로 `VehicleDebug`는,

- 차량 Pawn이 런타임/주행/입력 상태를 문자열로 조합하고,
- 로컬 플레이어용 위젯이 그 문자열을 화면에 띄우는
- 텍스트 기반 차량 진단 기능

이다.

이번 확장의 목적은 이 기능을 단순 텍스트 출력에서 끝내지 않고,

- 구조화된 디버그 데이터 기반으로 정리하고,
- Debug HUD와 Debug Panel을 함께 지원하며,
- 카테고리 증가에도 유지보수가 가능한 구조로 바꾸는 것

이다.

---

## 2. 현재 문제 정의

현재 구조는 시작점으로는 충분하지만, 앞으로 디버그 항목이 늘어나면 아래 문제가 커질 가능성이 높다.

### 2.1 문자열 중심 구조의 확장 한계

현재 핵심 출력물이 `FString` / `FText` 기반이다.

이 구조는 다음 상황에 약하다.

- 같은 데이터를 HUD와 Panel에서 다르게 보여줘야 할 때
- 값 기반 강조, 경고, 색상, 배지를 붙여야 할 때
- 특정 카테고리만 따로 숨기거나 정렬해야 할 때
- 일부 값은 숫자 위주, 일부는 이벤트 위주로 보여줘야 할 때

즉, 현재 구조는 정보의 원본이 아니라 이미 가공된 최종 출력물에 가깝다.

### 2.2 표시 레이어와 데이터 레이어 경계가 약함

현재 위젯은 얇은 표시 레이어이긴 하지만, 여전히 `GetDebugTextByDisplayMode()` 같은 텍스트 함수에 많이 의존한다.

이 상태에서 HUD와 Panel을 늘리면,

- 텍스트 포맷 함수가 계속 커지고,
- 표시 요구가 늘수록 문자열 조합 책임이 과도해지고,
- UI 종류별 재사용성이 낮아질 가능성이 크다.

### 2.3 Pawn 직접 위젯 생성 구조의 장기 한계

현재는 `BP_CFVehiclePawn`이 직접 위젯을 만들고 뷰포트에 붙인다.

이 구조는 현재 로컬 차량 디버그에는 맞지만, 앞으로 아래 요구가 생기면 한계가 있다.

- 디버그 HUD를 전역적으로 켜고 끄기
- Possess 대상이 바뀌어도 HUD 유지하기
- 차량 외 다른 디버그와 공존하기
- 전역 디버그 입력 체계와 연결하기

## 3. 이번 확장의 목표

이번 확장에서 반드시 달성해야 하는 목표는 아래와 같다.

### 3.1 구조화된 디버그 데이터 원본 확보

VehicleDebug의 원본 데이터는 문자열이 아니라 Snapshot 구조체여야 한다.

문자열은 유지할 수 있지만, 그것은 원본이 아니라 표시용 파생 출력이어야 한다.

### 3.2 Debug HUD와 Debug Panel 동시 지원

하나의 디버그 데이터 원본으로 아래 두 표시 형태를 모두 지원해야 한다.

- 항상 보이는 Compact HUD
- 카테고리 기반 상세 Panel

### 3.3 카테고리 확장 가능 구조 확보

처음에는 아래 4개 카테고리를 기본 기준으로 삼는다.

- Overview
- Drive
- Input
- Runtime

이후 필요 시 다음 카테고리 확장을 허용한다.

- Wheel
- Camera
- Owner
- Network
- Weapon
- AI

### 3.4 현재 텍스트 출력과의 호환 유지

기존 `SingleLine / MultiLine` 텍스트 출력은 당장 제거하지 않는다.

초기 단계에서는 아래 원칙으로 유지한다.

- 기존 텍스트 출력은 호환용 출력 모드로 유지
- 새 구조의 Snapshot을 문자열로 변환하는 방식으로 축소
- 이후에도 빠른 확인용 텍스트 모드는 남길 수 있음

### 3.5 현재 상태와 최근 이벤트 분리

디버그에는 아래 두 종류가 공존해야 한다.

- 현재 값
- 최근 변화 이벤트

예:
- 현재 속도, 현재 입력, 현재 상태
- 최근 상태 전이, 최근 런타임 재초기화, 최근 입력 주도권 변경

이 둘을 동일한 긴 문자열 한 덩어리로만 다루지 않도록 분리해야 한다.

## 4. 비목표

이번 설계 범위에서 바로 하지 않는 항목은 아래와 같다.

### 4.1 완전한 디버그 콘솔 시스템 구축

명령어 콘솔, 필터 검색, 외부 로그 브라우저 수준의 대형 도구는 이번 범위가 아니다.

### 4.2 그래프/차트 중심 시각화

속도 그래프, 입력 히스토리 곡선, 프레임 타임 차트 등은 후속 확장으로 미룬다.

### 4.3 에디터 전용 복합 디버그 툴 윈도우

Slate 기반 대형 툴 창이나 에디터 전용 디버그 도킹 패널은 이번 범위에 포함하지 않는다.

### 4.4 모든 차량 외 시스템 통합

이번 설계는 VehicleDebug 중심이다.
다른 시스템 디버그와의 전면 통합은 나중 단계다.

## 5. 기능 요구사항

## 5.1 데이터 요구사항

1. VehicleDebug 원본 데이터는 구조체 기반이어야 한다.
2. 카테고리별 하위 구조를 가질 수 있어야 한다.
3. 표시 레이어는 문자열이 아닌 필드 단위 데이터 접근이 가능해야 한다.
4. 수치값, enum, bool, 요약 문자열을 함께 다룰 수 있어야 한다.
5. 최근 이벤트 목록을 별도로 제공할 수 있어야 한다.

## 5.2 UI 요구사항

1. Compact HUD를 지원해야 한다.
2. 상세 Panel을 지원해야 한다.
3. 카테고리 단위 접기/펼치기 또는 표시/비표시가 가능해야 한다.
4. 표시 모드는 HUD와 Panel을 독립적으로 켜고 끌 수 있어야 한다.
5. 기존 Text 기반 표시도 유지 가능해야 한다.

## 5.3 운영 요구사항

1. 로컬 플레이어 기준으로만 표시되어야 한다.
2. 데이터 생성 책임과 표시 책임을 가능한 분리해야 한다.
3. 카테고리나 항목이 늘어나도 기존 HUD 전체를 갈아엎지 않아야 한다.
4. PIE에서 빠르게 확인 가능한 사용성을 유지해야 한다.

---

## 6. 품질 요구사항

### 6.1 가독성

- HUD는 한눈에 읽혀야 한다.
- Panel은 카테고리별로 의미가 구분되어야 한다.
- 같은 성격의 값은 같은 그룹에 있어야 한다.

### 6.2 유지보수성

- Drive 관련 항목 추가가 Input 관련 코드 변경으로 이어지지 않도록 책임을 분리한다.
- 문자열 포맷 함수 하나에 모든 항목이 몰리지 않도록 한다.
- 새 카테고리 추가 시 기존 UI의 핵심 구조를 크게 바꾸지 않도록 한다.

### 6.3 디버깅 효율

- 현재 값 확인 속도가 빨라야 한다.
- 최근 상태 변화 확인이 쉬워야 한다.
- 실패 원인 요약이 Runtime 카테고리 안에서 찾기 쉬워야 한다.

## 8. 핵심 결정 문장

이번 확장의 가장 중요한 결정은 아래 문장으로 고정한다.

> VehicleDebug의 원본은 문자열이 아니라 Snapshot 구조체이며, 기존 텍스트 출력은 그 Snapshot을 기반으로 만든 호환 출력 모드로 유지한다.

---

## 9. 성공 기준

이번 설계가 성공으로 판단되려면 아래 조건을 만족해야 한다.

1. 기존 텍스트 디버그가 깨지지 않는다.
2. 같은 원본 데이터로 HUD와 Panel이 모두 표시된다.
3. Overview / Drive / Input / Runtime이 명확히 분리된다.
4. 최근 이벤트를 현재 값과 분리해서 다룰 수 있다.
5. 이후 카테고리 추가가 가능한 구조가 된다.
