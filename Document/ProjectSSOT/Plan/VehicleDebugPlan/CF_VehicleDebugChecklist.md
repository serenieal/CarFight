# VehicleDebug v2 체크리스트

- Version: 0.1.0
- Date: 2026-04-22
- Status: Draft
- Scope: 설계/구현/검증 시 확인할 점검 목록

---

## 1. 사용 목적

이 문서는 VehicleDebug v2 작업 중,

- 설계를 빠뜨리지 않았는지
- 구현이 방향에서 벗어나지 않았는지
- 검증이 충분한지

점검하기 위한 체크리스트다.

---

## 2. 설계 체크리스트

### 2.1 원본 데이터 구조

- [ ] VehicleDebug 원본이 문자열이 아니라 Snapshot 구조체인지 확인했다.
- [ ] `Overview / Drive / Input / Runtime` 카테고리 경계를 정의했다.
- [ ] 현재 값과 최근 이벤트를 분리할 계획이 있는지 확인했다.
- [ ] 기존 `FCFVehicleDebugSnapshot`를 확장할지, 새 하위 구조를 둘지 결정했다.
- [ ] HUD와 Panel이 같은 원본 데이터를 공유하도록 설계했다.

### 2.2 책임 분리

- [ ] C++와 BP의 책임을 구분했다.
- [ ] 데이터 수집 책임이 UI 위젯으로 새지 않게 했다.
- [ ] 문자열 생성 함수가 원본 데이터 역할을 하지 않게 했다.
- [ ] 카테고리 추가가 한 거대한 포맷 함수 확장으로 이어지지 않게 했다.

### 2.3 표시 정책

- [ ] HUD와 Panel을 별도 표시 스위치로 관리할지 결정했다.
- [ ] Legacy Text View를 유지할지 결정했다.
- [ ] 카테고리별 표시/숨김 정책을 정했다.
- [ ] 로컬 플레이어 기준 표시 정책이 유지되는지 확인했다.

---

## 3. C++ 구현 체크리스트

### 3.1 Snapshot

- [ ] `Overview` 구조가 정의되었다.
- [ ] `Drive` 구조가 정의되었다.
- [ ] `Input` 구조가 정의되었다.
- [ ] `Runtime` 구조가 정의되었다.
- [ ] 각 구조의 필드명이 UI에서 바로 읽어도 이해 가능한 수준인지 확인했다.

### 3.2 기존 호환 유지

- [ ] `GetVehicleDebugSnapshot()`가 새 구조를 반영한다.
- [ ] `GetDebugTextSingleLine()`가 깨지지 않는다.
- [ ] `GetDebugTextMultiLine()`가 깨지지 않는다.
- [ ] `GetDebugTextByDisplayMode()`가 깨지지 않는다.
- [ ] `ShouldShowDebugWidget()` 동작이 기존 의도와 충돌하지 않는다.

### 3.3 이벤트

- [ ] 이벤트를 별도 구조로 분리할지 결정했다.
- [ ] 최소한 Drive 상태 전이 이벤트는 별도 처리 가능한지 확인했다.
- [ ] 이벤트 개수 제한 정책을 정했다.
- [ ] 이벤트 텍스트가 너무 길어지지 않게 했다.

---

## 4. 블루프린트 / UMG 체크리스트

### 4.1 HUD

- [ ] Compact HUD 위젯이 있다.
- [ ] HUD에 핵심값만 올렸다.
- [ ] HUD가 패널처럼 과도하게 커지지 않았다.
- [ ] 숫자값과 상태값이 한눈에 읽히는지 확인했다.

### 4.2 Panel

- [ ] 카테고리 블록 구조가 있다.
- [ ] 라벨과 값 정렬이 읽기 쉽다.
- [ ] `Overview / Drive / Input / Runtime` 구분이 명확하다.
- [ ] 긴 요약 문자열이 레이아웃을 깨지 않게 했다.
- [ ] 스크롤 또는 접기/펼치기 구조가 있다.

### 4.3 Legacy Text

- [ ] 기존 텍스트 위젯이 필요 시 유지된다.
- [ ] 텍스트 출력이 Snapshot 기반 파생 결과인지 확인했다.
- [ ] Text View가 HUD/Panel과 충돌하지 않는지 확인했다.

---

## 5. 사용성 체크리스트

- [ ] HUD만 봐도 현재 핵심 상태를 알 수 있다.
- [ ] Panel을 열면 원인 분석이 가능하다.
- [ ] Drive 문제와 Input 문제를 구분해서 읽을 수 있다.
- [ ] Runtime 실패 원인을 찾기 쉽다.
- [ ] 최근 상태 변화가 필요할 때 별도로 추적 가능하다.

---

## 6. 회귀 체크리스트

- [ ] 기존 텍스트 디버그가 사라지지 않았다.
- [ ] 로컬 제어 차량에서만 표시되는 정책이 유지된다.
- [ ] 기존 BeginPlay 생성 흐름을 바꿨다면 의도를 문서화했다.
- [ ] MultiLine 출력에서 중요 정보가 누락되지 않았다.
- [ ] 기존 QA/디버깅 사용 습관이 완전히 끊기지 않게 했다.

---

## 7. 성능 체크리스트

- [ ] 모든 값을 매 프레임 불필요하게 문자열화하지 않는다.
- [ ] HUD와 Panel이 서로 중복 계산을 하지 않는다.
- [ ] 이벤트 로그가 무한 증가하지 않는다.
- [ ] 불필요한 바인딩 과다 사용을 피했다.
- [ ] 값 변화가 거의 없는 항목은 과도하게 갱신하지 않도록 고려했다.

---

## 8. 문서화 체크리스트

- [ ] 현재 상태 문서와 설계 문서의 역할이 섞이지 않았다.
- [ ] 구현이 바뀌면 `VehicleDebug.md` 갱신 필요 여부를 확인했다.
- [ ] 새 위젯/새 토글/새 구조체 이름을 문서에 반영했다.
- [ ] 계획과 실제 구현 차이를 나중에 추적할 수 있게 남겼다.

---

## 9. 최종 완료 체크리스트

- [ ] Snapshot 중심 구조가 도입되었다.
- [ ] Compact HUD가 동작한다.
- [ ] Category Panel이 동작한다.
- [ ] 기존 Text View가 필요 시 유지된다.
- [ ] Overview / Drive / Input / Runtime이 분리되었다.
- [ ] 이후 카테고리 추가가 가능한 구조가 되었다.
